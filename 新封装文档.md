# 新封装文档.md

## 1. 重构目标

当前 `main.py` 存在明显的封装问题：

- 粒子系统、棍子系统、关卡系统、相机、音效、图片加载、地图数据全部写在 `main.py` 中。
- `Stick` 类直接依赖 `Level` 的具体实现。
- 墙壁只有 `"normal" | "fragile" | "goal"` 这种简单类型，无法支持“不稳定 + 移动 + 脆弱 + 终点”等组合特性。
- 地图直接写在主文件中，地图设计师无法独立维护关卡。
- 地图设计师仍然需要修改 Python 代码，内容制作成本较高。
- 墙壁、道具、危险物等实体没有统一对象模型。
- 渲染、逻辑、资源管理混在一起，后期扩展会越来越困难。

本次重构目标是：

```text
把 main.py 变成只负责游戏流程的入口文件，
把具体系统拆分到独立模块，
墙壁使用组合组件而不是继承，
关卡地图放入 maps/ 文件夹，
地图设计师只通过符号文本绘制关卡。
```

---

## 2. 核心设计原则

### 2.1 主文件只负责流程

`main.py` 不应该再包含：

- `Stick` 类
- `Level` 类
- `ParticleSystem`
- `SoundManager`
- `ImageManager`
- 地图数据
- 墙壁解析逻辑
- 道具解析逻辑
- 具体 UI 绘制逻辑

`main.py` 只保留：

- 初始化游戏
- 状态切换
- 输入分发
- 调用各系统更新
- 调用渲染系统
- 游戏主循环

---

### 2.2 使用组合代替继承

墙壁不能设计成：

```text
NormalWall
FragileWall
MovingWall
UnstableWall
FragileMovingWall
UnstableMovingWall
FragileUnstableMovingWall
GoalMovingUnstableWall
...
```

这种设计会导致类爆炸。

正确设计是：

```text
Wall
 ├── FragileComponent
 ├── MovingComponent
 ├── UnstableComponent
 ├── GoalComponent
 └── RenderComponent
```

一个墙壁对象可以同时拥有多个组件：

```text
一个墙壁 = 可移动 + 不稳定 + 脆弱 + 终点
```

例如：

```python
wall.components = {
    "fragile": FragileComponent(...),
    "moving": MovingComponent(...),
    "unstable": UnstableComponent(...),
    "goal": GoalComponent(...),
}
```

### 7.1 职责

把 Pygame 的输入事件转换为游戏内部事件。

例如：

```text
pygame.K_SPACE -> "space_pressed"
pygame.K_ESCAPE -> "quit"
pygame.K_F3 -> "toggle_debug"
```

### 7.2 示例

```python
class InputManager:
    def poll(self):
        events = []

        for event in pygame.event.get():
            if event.type == pygame.QUIT:
                events.append("quit")

            elif event.type == pygame.KEYDOWN:
                if event.key == pygame.K_SPACE:
                    events.append("space_pressed")
                elif event.key == pygame.K_ESCAPE:
                    events.append("quit")
                elif event.key == pygame.K_F3:
                    events.append("toggle_debug")

        return events
```

## 8. 事件总线 EventBus

文件位置：

```text
core/event_bus.py
```

### 8.1 为什么需要事件总线

现在代码里大量逻辑是：

```python
if result == "anchor_goal":
    播音效
    生成粒子
    改状态
    生成气泡
```

这会让 `Game` 越来越臃肿。

重构后可以让系统发布事件：

```python
event_bus.emit("anchor_success", data)
event_bus.emit("anchor_miss", data)
event_bus.emit("wall_break", data)
event_bus.emit("goal_reached", data)
```

然后：

- 音效系统监听事件播放音效
- 粒子系统监听事件生成粒子
- UI 系统监听事件生成气泡
- 游戏状态系统监听胜利/死亡事件

### 8.2 示例

```python
class EventBus:
    def __init__(self):
        self.listeners = {}

    def subscribe(self, event_name, callback):
        self.listeners.setdefault(event_name, []).append(callback)

    def emit(self, event_name, data=None):
        for callback in self.listeners.get(event_name, []):
            callback(data)
```

# 9.1 当前 Stick 类的问题

当前 `Stick` 类的问题：

1. 直接依赖 `Level`：

```python
wall_result = level.get_wall_at_point(...)
item_result = level.get_item_at_point(...)
```

这导致 `Stick` 和 `Level` 强耦合。

2. 只保存墙壁类型：

```python
self.attached_wall_type = "normal"
```

但新设计中墙壁可能同时是：

```text
不稳定 + 移动 + 脆弱 + 终点
```

所以不能只保存字符串类型。

3. 没有保存锚点在墙壁上的局部位置。

不稳定墙壁会旋转，所以锚点必须跟随墙壁旋转。  
因此棍子必须保存：

```python
attached_wall
attached_anchor_local_pos
```

4. 物理逻辑中没有考虑墙壁移动速度和墙壁旋转。

移动墙壁和不稳定墙壁会影响锚点位置。

- 棍子自身状态

- 端点计算
- 切换锚点
- 应用道具效果
- 强制脱锚
- 记录当前抓住的墙壁对象

`Stick` 不负责：

- 遍历墙壁列表
- 遍历道具列表
- 判断胜利
- 播放音效

# 9.6 Stick 的端点计算

```python
def get_direction(self):
    return Vector2(math.cos(self.angle), math.sin(self.angle))

def get_endpoints(self):
    d = self.get_direction()
    half = self.length / 2

    if self.state == "airborne":
        left = self.center + d * half
        right = self.center - d * half
        return left, right

    if self.state == "anchored":
        anchor = self.anchor_world_pos

        if self.anchor_side == "left":
            left = anchor
            right = anchor - d * self.length
        else:
            right = anchor
            left = anchor + d * self.length

        return left, right
```

注意：  
这里的方向定义要在项目中统一。  
如果你希望 `left = center - d * half`，也可以，但必须全项目一致。

# 9.8 switch_anchor 新逻辑

```python
def switch_anchor(self, anchor_query):
    old_anchor_side = self.anchor_side
    new_anchor_side = self.get_opposite_side()

    new_anchor_pos = self.get_endpoint_position(new_anchor_side)

    wall = anchor_query.find_wall_at_point(
        new_anchor_pos,
        self.get_endpoint_radius(new_anchor_side)
    )

    # 先脱离旧墙壁
    self.detach_from_current_wall()

    self.anchor_side = new_anchor_side

    if wall is not None:
        self.attach_to_wall(wall, new_anchor_pos)

        if wall.has_component("goal"):
            return StickEvent("goal", wall=wall, position=new_anchor_pos)

        if wall.has_component("unstable"):
            return StickEvent("anchor_unstable", wall=wall, position=new_anchor_pos)

        if wall.has_component("fragile"):
            return StickEvent("anchor_fragile", wall=wall, position=new_anchor_pos)

        return StickEvent("anchor_success", wall=wall, position=new_anchor_pos)

    else:
        self.become_airborne_from_current_motion()
        return StickEvent("anchor_miss", position=new_anchor_pos)
```

```python
def update_airborne(self, dt, gravity):
```

# 10.5 Wall 坐标转换

墙壁需要支持旋转，所以必须提供：

```python
world_to_local(point)
local_to_world(point)
contains_point(point, radius)
```

```python
def world_to_local(self, point):
    p = point - self.position
    return rotate_vector(p, -self.angle)

def local_to_world(self, local):
    return self.position + rotate_vector(local, self.angle)

def contains_point(self, point, radius=0):
    local = self.world_to_local(point)

    return (
        abs(local.x) <= self.length / 2 + radius
        and abs(local.y) <= self.thickness / 2 + radius
    )
```

### 12.1 职责

- 当锚点离开墙壁后，墙壁碎裂。
- 可以配置是否立即碎裂，是否播放碎裂事件。

```python
class FragileComponent(WallComponent):
    def __init__(self, break_on_detach=True):
```

# 16.5 根据抓住位置计算力矩

```python
def compute_torque(self):
    if self.attached_stick is None or self.anchor_local_pos is None:
        return 0.0

    half_length = self.owner.length / 2

    offset = self.anchor_local_pos.x / max(half_length, 0.0001)
    offset = clamp(offset, -1.0, 1.0)

    stick_weight = self.attached_stick.total_mass * 9.8

    torque = offset * stick_weight * self.torque_scale
    torque = self.apply_direction_limit(torque)

    return torque
```

---

# 16.6 倾斜方向限制

```python
def apply_direction_limit(self, torque):
    if self.tilt_direction == "both":
        return torque

    if self.tilt_direction == "clockwise":
        return max(0.0, torque)

    if self.tilt_direction == "counter_clockwise":
        return min(0.0, torque)

    return torque
```

注意：  
Pygame 坐标系中 y 轴向下，顺逆时针正负号可能需要根据实际效果调整。

# 16.7 update

```python
def update(self, dt):
    if self.attached_stick is not None:
        torque = self.compute_torque()

        self.angular_acceleration = torque / max(self.inertia, 0.0001)
        self.angular_velocity += self.angular_acceleration * dt

    else:
        self.angular_acceleration = 0.0

        if self.return_speed > 0:
            self.tilt_angle = move_towards(
                self.tilt_angle,
                0.0,
                self.return_speed * dt
            )

    self.angular_velocity -= self.angular_velocity * self.damping * dt

    self.tilt_angle += self.angular_velocity * dt

    if self.tilt_angle > self.max_tilt_angle:
        self.tilt_angle = self.max_tilt_angle
        self.on_reach_max_tilt()

    elif self.tilt_angle < -self.max_tilt_angle:
        self.tilt_angle = -self.max_tilt_angle
        self.on_reach_max_tilt()

    self.owner.angle = self.tilt_angle
```

## 20. 地图文件夹设计：使用 Tiled Map Editor

地图不再保存在 `main.py` 中，也不再使用纯字符串地图描述关卡。

由于本游戏中的关卡对象具有较复杂的参数，例如：

- 墙壁长度
- 墙壁厚度
- 墙壁旋转角度
- 是否脆弱
- 是否不稳定
- 是否移动
- 是否是终点
- 移动路径
- 移动方式
- 移动速度
- 不稳定墙壁最大倾斜角
- 不稳定墙壁支点位置
- 是否达到最大倾斜角后破坏

因此正式关卡采用：

```text
Tiled Map Editor + 对象层 + 自定义属性
```

而不是：

```text
map.txt + 字符符号 + meta.json 手动绑定移动路径
```

---

# 20.1 每个关卡一个文件夹

每个关卡放在 `maps/` 目录下的独立文件夹中：

```text
maps/
    level1/
        level.tiled.json
        level.json
        preview.png

    level2/
        level.tiled.json
        level.json
        preview.png
```

文件说明：

| 文件 | 作用 | 谁维护 |
|---|---|---|
| `level.tiled.json` | Tiled 原始导出文件 | 地图设计师 |
| `level.json` | 游戏实际读取的关卡文件 | 工具自动生成 |
| `preview.png` | 关卡预览图，可选 | 地图设计师 / 美术 |

推荐流程：

```text
地图设计师在 Tiled 中编辑 level.tiled.json
↓
运行转换工具 tiled_converter.py
↓
生成游戏使用的 level.json
↓
游戏通过 LevelLoader 读取 level.json
```

也可以在初期直接让游戏读取 `level.tiled.json`。  
但更推荐转换为自己的 `level.json`，这样游戏运行时的数据结构更干净。

---

# 20.2 为什么使用 Tiled

Tiled Map Editor 适合本项目的原因：

1. **支持可视化拖拽编辑。**
2. **支持矩形对象，可以直接表示墙壁长度和厚度。**
3. **支持对象自定义属性，可以配置脆弱、移动、不稳定等参数。**
4. **支持 Polyline，可以表示移动路径。**
5. **支持多个对象层，方便区分墙壁、道具、危险物、路径、出生点。**
6. **地图设计师不需要写 Python。**
7. **地图设计师不需要手写坐标数组。**
8. **复杂墙壁不需要用字符硬编码。**

之前的字符地图方案适合非常早期的原型，但不适合本游戏的正式关卡制作。

各层职责如下：

| 图层 | 内容 |
|---|---|
| `Player` | 玩家出生点 |

| `Hazards` | 危险物 |
| `Paths` | 移动墙路径 |
| `Triggers` | 触发区域，可选 |

# 22.2 移动路径坐标规则

Tiled 的 Polyline 点通常是相对于路径对象自身位置的局部坐标。

# 22.3 移动路径起点规则

推荐规则：


```python
def convert_polyline_path(obj):
    base_x = obj.get("x", 0)
    base_y = obj.get("y", 0)

    points = []

    for p in obj.get("polyline", []):
        points.append([
            base_x + p.get("x", 0),
            base_y + p.get("y", 0)
        ])

    return points
```

# 27.3 LevelLoader 示例代码结构

```python
class LevelLoader:
    @staticmethod
    def load(level_id):
        path = f"maps/{level_id}/level.json"

        data = load_json_file(path)

        level = Level({
            "level_id": data["level_id"],
            "name": data["name"],
            "world": data["world"],
            "background": data.get("background"),
            "music": data.get("music"),
            "player_start": data["player_start"],
            "lava": data["lava"],
        })

        for obj in data.get("objects", []):
            if obj["type"] == "wall":
                wall = ObjectFactory.create_wall(obj)
                level.walls.append(wall)

            elif obj["type"] == "item":
                item = ObjectFactory.create_item(obj)
                level.items.append(item)

            elif obj["type"] == "hazard":
                hazard = ObjectFactory.create_hazard(obj)
                level.hazards.append(hazard)

        return level
```

## 28. ObjectFactory 对象工厂

文件位置：

```text
level/object_factory.py
```

由于现在关卡对象来自 `level.json`，推荐用 `ObjectFactory` 统一创建对象。

```python
class ObjectFactory:
    @staticmethod
    def create_wall(obj):
        wall = Wall({
            "id": obj["id"],
            "position": [obj["x"], obj["y"]],
            "width": obj["width"],
            "height": obj["height"],
            "rotation": obj.get("rotation", 0),
            "prefab": obj.get("prefab", "normal_wall"),
            "appearance": obj.get("appearance"),
        })

        components = obj.get("components", {})

        for name, cfg in components.items():
            component = ComponentFactory.create(name, cfg)
            wall.add_component(name, component)

        return wall

    @staticmethod
    def create_item(obj):
        return Item({
            "id": obj["id"],
            "prefab": obj["prefab"],
            "position": [obj["x"], obj["y"]],
        })

    @staticmethod
    def create_hazard(obj):
        return Hazard({
            "id": obj["id"],
            "prefab": obj["prefab"],
            "position": [obj["x"], obj["y"]],
            "width": obj.get("width", 32),
            "height": obj.get("height", 32),
        })
```

## 46. 最终设计总结

重构后的架构核心是：

```text
主流程归 Game，
地图归 maps，
加载归 LevelLoader，
实体归 entities，
墙壁能力归 components，
移动方式归 movement，
物理工具归 physics，
资源归 systems，
UI 特效归 render.py。
```

墙壁不再通过继承区分类型，而是通过组件组合能力：

```text
Wall + FragileComponent
Wall + MovingComponent
Wall + UnstableComponent
Wall + GoalComponent
Wall + MovingComponent + UnstableComponent + FragileComponent
```

这样可以避免类爆炸，并且让关卡设计更灵活。

地图不再写在 `main.py` 中，而是放到：

```text
maps/level1/map.txt
maps/level1/meta.json
```

地图设计师只需要用符号绘制关卡，不需要写 Python 代码。

棍子类也要从旧的“只记录墙壁类型”改成“记录墙壁对象和墙壁局部锚点”，这样才能支持：

- 移动墙壁
- 不稳定倾斜墙壁
- 脆弱墙壁
- 组合型墙壁

最终目标是让游戏系统变成：

```text
玩法逻辑清晰
关卡制作简单
墙壁能力可组合
UI 和特效可独立调整
main.py 足够干净
```
